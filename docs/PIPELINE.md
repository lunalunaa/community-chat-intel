# Stream Processing Pipeline — Architecture & Operations

> **Scope:** Documents the multi-stream analysis pipeline (`stream_b`, `stream_c`, `stream_d`, synthesis, post-analysis) architecture. All concrete numbers, dates, and the example community name below are from **one real example run** (a 28,005-message, 20-day Feishu chat export analyzing a Chinese-language user community for an AI coding-agent product) — kept in place because seeing genuine figures makes the architecture concrete. Swap in your own dataset's numbers and product name when you run this against a different community.  
> **Related:** The core keyword-analysis pipeline (`analyze.py`, `topics.py`, `crosstabs.py`) is documented in [README.md](./README.md). The research methodology rationale is in [plan.md](./plan.md).

---

## Architecture Overview

The pipeline runs four parallel **streams** against the same raw chat export, then synthesizes their outputs into a final report. Each stream answers different research questions using different technical approaches — no single method is reliable enough on its own.

```
                        pages.jsonl (raw Feishu export, NDJSON)
                                  │
                                  │  Stream A (topics.py)
                                  │  ─────────────────────────
                                  │  LLM topic classification
                                  │  12 categories × 24,545 msgs
                                  │  → topics.json
                                  │
                    ┌─────────────┼─────────────┐
                    │             │             │
              Stream B       Stream C       Stream D
              ─────────     ─────────     ─────────
              Semantic     Structured    Deterministic
              retrieval    fact extract  Python analytics
              BGE-M3 +     MiMo v2.5     (no LLM, free)
              FAISS +      per 100-msg   Pareto, stickiness,
              MiMo v2.5    chunks        gateway mix, reply
              18 queries   6,571 facts   graph, temporal
                    │             │             │
                    └─────────────┼─────────────┘
                                  │
                    cross_stream_synthesis.py
                    (MiMo v2.5-Pro, single 36K-char prompt)
                                  │
                        findings_final_v2.md
                                  │
                    build_final_report.py
                    (MiMo v2.5-Pro recommendations)
                                  │
                          final_report.md
                                  │
                    post_analysis.py
                    (brand audit, token drill, CSVs)
```

**Why four streams?** Each compensates for another's blind spots:

| Stream | Strength | Blind spot |
|--------|----------|------------|
| **A** (topics) | Broad coverage; every message tagged | Coarse; can't catch sub-thread nuance |
| **B** (semantic) | Finds things keyword search misses; narrative synthesis | Query-dependent; misses if you don't ask |
| **C** (facts) | Structured, queryable, counts everything | LLM hallucination risk; schema bleed |
| **D** (deterministic) | Ground truth; no hallucination risk | Surface-level; can't interpret sentiment |

## Prerequisites

All streams assume:

1. **Raw chat export** as NDJSON at a known path (one JSON object per line). For Feishu/Lark, produced by `@larksuite/cli v1.0.18` paginating `GET /open-apis/im/v1/messages` at 50/page.
2. **An LLM CLI configured** with a working provider for LLM streams (A, B, C, synthesis). The scripts shell out to `hermes chat -q` by default — no direct API keys needed in this codebase. Swap the `cmd` list in each script if you use a different CLI.
3. **Python 3.12+** with `sentence-transformers`, `faiss-cpu`, `numpy` for Stream B embeddings.
4. **Salt file** for user-ID hashing (auto-generated by `analyze.py` on first run; default `./user_hash_salt.key`, override via `SALT_FILE`).

## Stream B — Semantic Retrieval + Narrative Synthesis

**File:** `stream_b_embed_retrieve.py` (204 lines)

### What it does

1. Embeds all non-system, non-trivial messages with `BAAI/bge-m3` (multilingual, works on Chinese + English)
2. Builds a FAISS `IndexFlatIP` (inner product = cosine similarity on normalized vectors)
3. Runs 18 pre-written retrieval queries, each returning the top 30 most semantically-similar messages
4. For each query, feeds those 30 messages to MiMo v2.5 to produce a ~150-200 word narrative synthesis

### Inputs

| Input | Source |
|-------|--------|
| `pages.jsonl` | Raw Feishu export (NDJSON) |
| Salt key | `~/nous-chinese-analysis/user_hash_salt.key` |

### Outputs

| File | Description |
|------|-------------|
| `stream_b/embeddings.npy` | ~95 MB float32 embedding matrix |
| `stream_b/ids.json` | Message-ID ordering for the embedding array |
| `stream_b/retrieval_results.json` | Top 30 hits per query with scores |
| `stream_b/synthesized_findings.json` | Per-query narrative analyses (structured JSON) |
| `stream_b/findings.md` | Human-readable markdown version |

### The 18 queries

Grouped by research question:

| Category | Queries |
|----------|---------|
| **Messaging adapter demand** | `feishu_adapter_demand`, `feishu_actively_building`, `wechat_adapter_demand`, `dingtalk_adapter_demand` |
| **Gateway/provider landscape** | `nous_portal_usage`, `direct_vendor_api_usage`, `openrouter_usage`, `agent_key_sharing` |
| **Topic texture** | `install_friction`, `brand_identity_confusion`, `pricing_complaints`, `vpn_friction`, `success_stories`, `feature_requests`, `skills_memory_cron_usage` |
| **Competitor sentiment** | `comparison_with_claw_products`, `comparison_with_claude_code` |
| **External communities** | `mentions_of_external_communities` |

### Performance

- Embedding 28K messages: ~2-5 minutes on CPU (single-threaded)
- 18 synthesis calls: ~10-15 minutes sequential; each call is independent
- Embedding cache: if `embeddings.npy` + `ids.json` exist and match, skip re-embedding

### Gotchas

- **Hardcoded paths** — edit `CHAT_JSONL` and `OUT_DIR` at the top of the file before running
- **Embedding cache** — delete `stream_b/embeddings.npy` to force re-embedding after changing the input
- **MiMo availability** — synthesis calls use `hermes chat -q --provider nous --model xiaomi/mimo-v2.5`. If MiMo is unavailable, edit the `subprocess.run` command
- **Message truncation** — only first 1,500 chars of each message are embedded (to stay within BGE-M3's context window)

## Stream C — Structured Fact Extraction

**Files:** `stream_c_fact_extract.py` (221 lines), `stream_c_retry.py` (243 lines)

### What it does

1. Splits all content messages into windows of ~100 consecutive messages (preserving conversational order)
2. Feeds each chunk to MiMo v2.5 with a JSON schema requesting 10 categories of structured facts
3. Aggregates facts across all chunks
4. On failure, `stream_c_retry.py` re-runs failed chunks with tolerant JSON parsing

### The 10 fact categories

| Category | Output field | Example facts |
|----------|-------------|---------------|
| Install problems | `install_problems` | "WSL Docker can't mount", "pip install fails on Windows" |
| Provider usage | `provider_usage` | "using direct_kimi, sentiment=positive" |
| Messaging intent | `messaging_intent` | "wants_adapter for feishu" |
| Brand confusion | `brand_confusion` | "is example-product.org.cn official?" |
| API key sharing | `api_key_sharing_evidence` | "offering sk-key invite code" |
| Pricing complaints | `pricing_complaints` | "this product burns more tokens than a competitor" |
| Feature requests | `feature_requests` | "want one-click Feishu integration" |
| Success stories | `success_stories` | "built a WeChat bot with skills + cron" |
| VPN/network friction | `vpn_network_friction` | "can't access without VPN, need domestic mirror" |
| Competitor mentions | `competitor_mentions` | "Competitor A has better Feishu integration" |

### Inputs

| Input | Source |
|-------|--------|
| `pages.jsonl` | Raw Feishu export |
| Salt key | `~/nous-chinese-analysis/user_hash_salt.key` |

### Outputs

| File | Description |
|------|-------------|
| `stream_c/chunks_cache.jsonl` | Per-chunk raw LLM output (resume-safe; append-only) |
| `stream_c/aggregated_facts.json` | All 6,571 facts across 10 categories |
| `stream_c/summary.json` | Aggregate counts + provider/messaging/competitor breakdowns |
| `stream_c/extraction_errors.json` | Chunks that failed JSON parsing (for retry) |

### Execution flow

```
stream_c_fact_extract.py
  ├── Load messages, sort chronologically
  ├── Chunk into windows of 100
  ├── For each chunk (concurrency=4):
  │     ├── Check cache → skip if done
  │     ├── Call MiMo v2.5 with schema prompt
  │     ├── Parse JSON response
  │     ├── Write to cache (append, flush)
  │     └── Log progress every 5 chunks
  ├── Aggregate all facts
  └── Write aggregated_facts.json + summary.json

If extraction_errors.json is non-empty:
stream_c_retry.py
  ├── Reread error chunks
  ├── Retry with tolerant JSON parsing:
  │     ├── Strict json.loads
  │     ├── Strip trailing commas (MiMo's main failure mode)
  │     ├── json5 fallback
  │     └── Last-resort: extract last JSON object from concatenated output
  ├── Append recovered chunks to cache
  └── Rebuild aggregates
```

### Performance

- 244 chunks × ~45s each (at concurrency=4): ~45 minutes total
- Cache is append-only and survives Ctrl-C — rerun picks up where it left off
- First pass succeeded on 228/244 chunks; 16 failures recovered via retry

### Gotchas

- **Trailing commas** — MiMo v2.5 frequently adds trailing commas in JSON arrays. The retry script handles this; the main script does not (failures go to `extraction_errors.json`)
- **Schema bleed** — facts occasionally land in the wrong category (e.g., provider info in `api_key_sharing_evidence`). Aggregates are trustworthy; per-record field-type purity is not
- **Hardcoded paths** — edit `CHAT_JSONL` and `OUT_DIR` at the top
- **Salt file** — independently reads `user_hash_salt.key` (doesn't import `analyze.py`)

## Stream D — Deterministic Python Analytics

**Files:**
- `stream_d_analysis.py` (297 lines) — original version
- `stream_d_v2.py` — membership-aware (joined-via-system-message reconciliation)
- `stream_d_v3.py` — correct poster-vs-member reconciliation (cross-tenant user handling)
- **`stream_d_v4.py`** (223 lines) — **final, ground-truth-anchored version**

### What it does (v4)

Runs 6 pure-Python analyses with no LLM calls — instant, free, and hallucination-proof:

| Analysis | Output | What it measures |
|----------|--------|-----------------|
| **User Pareto** | `user_pareto_v4.json` | Message concentration: top-N posters' share of total messages, anchored to ground-truth membership |
| **User stickiness** | `user_stickiness_v4.json` | Retention breakdown: one-time posters vs single-day vs few-day vs multi-week (7+ days) active, as % of both posters and total human membership |
| **Temporal growth** | `temporal_growth_v4.json` | Day-by-day: new members joined, first-time human posters, human messages, active humans, cumulative curves |
| **Gateway mix** | `gateway_mix.json` | URL-based classification of which API gateways/providers users reference (a hosted service, direct vendor APIs, OpenRouter, self-hosted, proxy resellers) |
| **Reply graph** | `reply_graph.json` | Thread depth stats: total replies, distinct threads, threads with 5+/10+ replies, longest threads |
| **Top users HTML** | `top_20_users.html` | Readable full-message timelines of the 20 most active posters (for qualitative review) |

### Why v4 (ground-truth anchoring)

v1-v3 derived membership from system messages (join events), but Feishu doesn't log leave/removal events. This caused membership overcounts (3,168 observed join names vs 3,119 actual human members).

v4 fixes this by pinning to **live membership verified via Feishu UI** on 2026-04-23:
- 3,119 humans + 5 bots = 3,124 live members
- 878 humans ever posted (28.2% of humans)
- 2,241 silent lurkers (71.8%)
- ~49 inferred departures (3,168 joins − 3,119 current)

This is now the authoritative denominator for all retention and engagement rates.

### Gateway URL classifier

URLs in message content are matched against domain patterns to classify gateway choice:

| Pattern | Label |
|---------|-------|
| `api.your-hosted-service.example.com` | `hosted_service` |
| `api.moonshot.cn`, `api.kimi.com` | `direct_moonshot` |
| `api.z.ai`, `open.bigmodel.cn` | `direct_zhipu` |
| `api.minimaxi.com/io/chat` | `direct_minimax` |
| `ark.cn-beijing.volces.com`, `volcengine.com` | `direct_volcengine` |
| `api.deepseek.com` | `direct_deepseek` |
| `dashscope.aliyuncs.com` | `direct_dashscope_qwen` |
| `api.anthropic.com` | `direct_anthropic` |
| `api.openai.com` | `direct_openai` |
| `localhost:11434`, `ollama` | `self_hosted_ollama` |
| `aigc.green` | `shadow_docs_aigc_green` |
| `szygumin.icu`, `opennana.com`, `aitokenwave.com` | `proxy_reseller` |

### Performance

- All 6 analyses combined: <5 seconds on the full 28K-message corpus
- No API calls, no cost, no rate limits

### Gotchas

- **Hardcoded paths** — edit `CHAT_JSONL` and `OUT_DIR` at the top
- **Ground truth** — `GROUND_TRUTH_HUMANS` and `GROUND_TRUTH_BOTS` are hardcoded from Feishu UI verification. Must update for a different chat
- **Gateway classifier** — matches URLs in message text only. Users who talk about providers without pasting URLs won't show up here (Stream C covers those)
- **Reply graph** — only tracks explicit `reply_to` threading. Inline `@`-mentions and adjacent-message answers are invisible

## Cross-Stream Synthesis

**File:** `cross_stream_synthesis.py` (159 lines)

### What it does

Gathers outputs from **all four streams** + ground-truth anchors, builds a single ~36K-character context prompt, and feeds it to **MiMo v2.5-Pro** (the stronger Pro variant) to produce `findings_final_v2.md` — a 14-section, 4,000-6,000 word narrative survey.

### Sections produced

| # | Section | Primary source |
|---|---------|---------------|
| 1 | Executive Summary | All streams |
| 2 | Who Are They? | Stream D (membership ground truth) |
| 3 | What Do They Talk About? | Stream A (topic distribution) |
| 4 | Provider / Gateway Landscape | Stream C (provider_usage facts) |
| 5 | Messaging Platform Usage | Stream B (feishu/wechat queries) + Stream C |
| 6 | Install & Setup Reality | Stream C (install_problems) |
| 7 | Competitor Landscape | Stream C (competitor_mentions) + brand audit |
| 8 | Brand Identity & Confusion | Stream C (brand_confusion) |
| 9 | API Key Sharing / Gray Market | Stream C (api_key_sharing_evidence) |
| 10 | Pricing / Token Consumption | Stream C (pricing_complaints) |
| 11 | Network / VPN / Geographic Friction | Stream C (vpn_network_friction) |
| 12 | Feature Requests & Success Stories | Stream C |
| 13 | Help Ecosystem Health | Stream D (reply graph) |
| 14 | Notable Quotes / Texture | Sampled exemplars from Stream C |

### Model selection

The synthesis and recommendations use **MiMo v2.5-Pro** — the Pro variant costs more but produces notably better reasoning for long-form synthesis. The base MiMo v2.5 is used for per-message/chunk classification (Streams A, B, C) where throughput matters more than nuance.

### Gotchas

- **Context length** — the prompt is ~36K characters. If adding more data, watch for token limits
- **MiMo availability** — the Pro variant may not always be available. Falls back to base MiMo v2.5 if needed (edit the `--model` flag)
- **Hardcoded paths** — `BASE` is hardcoded at top

## Build Final Report

**File:** `build_final_report.py` (350 lines)

### What it does

Assembles `final_report.md` from multiple components:

1. **Front matter** — header with metadata
2. **Executive Summary** — extracted from `findings_final_v2.md` §1
3. **Methodology** — written directly in the script (hardcoded prose describing data collection, ground-truth anchoring, analysis streams, tools, privacy)
4. **Findings** — body of `findings_final_v2.md` (§§2-14, renumbered)
5. **Recommendations** — generated by a separate MiMo v2.5-Pro call with explicit P0/P1/P2/Watch tiering instructions and structural context
6. **Limitations & Caveats** — written directly in the script (sample limitations, instrument limitations, model/inference limitations, interpretation caveats)
7. **Appendices** — file inventory with sizes, reproduction instructions

### The recommendations generator

The recommendations prompt is architecturally significant — it explicitly tells the model:
- The structural context relevant to your product (e.g. does a platform-owner also own a competing product on that same platform, does an existing feature exist but isn't discoverable, is a gray-market API-key economy normalized, etc.)
- To produce P0/P1/P2/Watch tiers with cost/effort/impact estimates
- To include a section on "What Is Explicitly NOT Recommended"
- To include open questions for leadership

This ensures the recommendations are grounded in the data rather than generic best-practices.

## Post-Analysis

**File:** `post_analysis.py` (191 lines)

### What it does

Three standalone drill-downs that run after the main synthesis:

#### (a) Brand Audit
- Regex-scans all 28K messages against 23 brand/variant patterns
- Discovers patterns not already known (e.g., an unofficial shadow-docs cluster, or newly-surfaced competitor/clone names)
- Merges with Stream C `brand_confusion` incidents for resolution breakdown
- Output: `post/brand_audit.md` with counts, observations, and sampled messages

#### (b) Token Consumption Drill
- Keyword-filters for cost/consumption complaints specifically about Hermes
- Pulls Stream C `pricing_complaints` incidents mentioning Hermes
- Output: `post/token_consumption.md`

#### (c) CSV Exports
- Exports Stream C structured facts as UTF-8 BOM CSVs (Excel-compatible)
- One CSV per fact category: `install_problems.csv`, `provider_usage.csv`, `messaging_intent.csv`, `brand_confusion.csv`, `feature_requests.csv`, `success_stories.csv`, `pricing_complaints.csv`, `api_key_sharing_evidence.csv`, `competitor_mentions.csv`
- Output: `post/*.csv`

## Full Execution Order

When reproducing against a new dataset, run in this order:

```bash
# 0. Prerequisites — get the raw export
#    (Feishu: use @larksuite/cli; Discord: use DiscordChatExporter)
#    Result: pages.jsonl (NDJSON, one JSON object per line)

# 1. Core pipeline (optional — establishes keyword baselines)
python3 analyze.py --input pages.jsonl --platform lark --out ./out -v

# 2. Stream A — topic classification
python3 topics.py \
    --input-chat pages.jsonl \
    --platform lark \
    --out ./out/topics.json

# 3. Stream D — deterministic analysis (run first; cheapest, catches data issues)
python3 stream_d_v4.py
#    ↑ edit CHAT_JSONL, OUT_DIR, GROUND_TRUTH_HUMANS/BOTS before running

# 4. Stream B — semantic retrieval + synthesis
python3 stream_b_embed_retrieve.py
#    ↑ edit CHAT_JSONL, OUT_DIR before running
#    Slowest step (embedding). Cache survives reruns.

# 5. Stream C — structured fact extraction
python3 stream_c_fact_extract.py
#    ↑ edit CHAT_JSONL, OUT_DIR before running
#    If extraction_errors.json is non-empty:
python3 stream_c_retry.py

# 6. Cross-stream synthesis
python3 cross_stream_synthesis.py
#    → findings_final_v2.md

# 7. Post-analysis drills
python3 post_analysis.py
#    → post/brand_audit.md, post/token_consumption.md, post/*.csv

# 8. Final report assembly
python3 build_final_report.py
#    → final_report.md
```

## Adapting for a New Dataset

All stream scripts have hardcoded paths at the top. Before running:

1. **Set `CHAT_JSONL`** — path to the raw NDJSON export
2. **Set `OUT_DIR`** — output directory (creates `stream_b/`, `stream_c/`, `stream_d/`, `post/` subdirectories)
3. **Set ground truth** (Stream D v4 only) — `GROUND_TRUTH_HUMANS` and `GROUND_TRUTH_BOTS` from platform UI
4. **Verify salt file** — all scripts read `~/nous-chinese-analysis/user_hash_salt.key`
5. **Check model availability** — scripts call `hermes chat -q --provider nous --model xiaomi/mimo-v2.5` (or `-pro`). If MiMo is down, edit the `subprocess.run` command to use a different model

### Minimum changes for a new dataset

```python
# In each stream script, edit these lines at the top:
CHAT_JSONL = Path("/path/to/your/pages.jsonl")
OUT_DIR = Path("/path/to/your/out_full")

# In stream_d_v4.py, also edit:
GROUND_TRUTH_HUMANS = 1234   # from platform UI
GROUND_TRUTH_BOTS = 0        # from platform UI
```

## Key Design Decisions

### Why shell out to `hermes chat -q` instead of using APIs directly?

- **Dogfooding:** if your product is itself an AI-agent tool, you can analyze its own users by using the product itself as the LLM orchestration layer
- **Zero auth surface:** No API keys in the scripts — all provider auth lives in your LLM CLI's own config
- **Provider portability:** Switch models by changing one `--model` flag
- **Rate limiting, retries, streaming:** Handled by the CLI, not reimplemented per-script

### Why Stream D v4 anchors to live UI membership?

Earlier versions derived membership from system-message join events. But Feishu doesn't log leave/removal events, so join-event counts are always an overcount. The live UI figure (3,119 humans) is the only reliable denominator for retention rates.

### Why Stream C uses per-chunk caching (append-only JSONL)?

The 244-chunk extraction takes ~45 minutes. If interrupted (Ctrl-C, network blip, MiMo outage), the cache ensures only unprocessed chunks are rerun. The append-only design means even partial progress is never lost.

## Daily Monitoring Service

**File:** `daily_service.py` (1,500 lines) — automated daily fetch, analysis, digest, and dashboard.

### What it does

A standalone service that runs daily (via cron) to:

1. **Fetch** new messages from the Feishu group via `@larksuite/cli` (incremental since last run)
2. **Analyze** keyword extraction, language detection, question detection, friction signals
3. **Digest** a structured daily summary generated by MiMo v2.5 (one LLM call per day)
4. **Dashboard** a self-contained HTML page with Chart.js activity charts

### Commands

```bash
python3 daily_service.py --init          # First-time setup
python3 daily_service.py --run           # Fetch + analyze + digest + dashboard
python3 daily_service.py --run --mock    # Test with stored messages (no live fetch)
python3 daily_service.py --status        # Show DB state, recent activity
python3 daily_service.py --digest 3      # Show last 3 digests in terminal
python3 daily_service.py --dashboard     # Regenerate dashboard from DB
python3 daily_service.py --config        # Show/set config (chat_id, member_count, etc.)
```

### State (SQLite)

Single database at `daily_state.db`:

| Table | Purpose |
|-------|---------|
| `config` | `chat_id`, `chat_name`, `member_count`, `timezone` |
| `state` | `last_fetch_ts`, `total_messages_fetched`, `last_run_at` |
| `messages` | All seen messages (message_id PK, timestamp, sender_hash, content, keywords) |
| `digests` | Per-day summaries (markdown, TL;DR, counts, friction, unanswered) |
| `daily_stats` | Per-day aggregate numbers (for dashboard charts) |

### Digest format

Each daily digest is a structured markdown with:
- **Header** — date, message count, active users, day-over-day delta
- **What Happened** — 3-5 bullet points capturing the main discussions
- **Key Themes** — table with topic, share, and signal interpretation
- **Friction & Issues** — problems, complaints, errors
- **Notable** — long threads, resolved questions, competitor mentions, brand confusion
- **TL;DR** — one-sentence summary for notifications

Digests are saved to `dailies/YYYY-MM-DD.md` and appended to a rolling `dailies/feed.md` (last 7 days).

### Dashboard

`dailies/dashboard.html` — self-contained, no server required. Open in any browser.

Features:
- **KPIs** — today's messages, active users, Chinese share, questions, friction signals
- **Activity chart** — 14-day bar chart (Chart.js)
- **User chart** — 14-day active-user line chart
- **Friction chart** — friction + questions trend lines
- **Top keywords** — per-category keyword breakdown
- **Topic distribution** — table with proportional bar visualization
- **Latest digest** — full daily digest rendered inline
- **Recent digests** — last 7 days at a glance

Dark theme (GitHub-style), responsive, loads Chart.js from CDN.

### Scheduling

```bash
# 1. Create a thin wrapper script (cron --script needs files under ~/.hermes/scripts/)
cat > ~/.hermes/scripts/chinese-community-daily.sh << 'EOF'
#!/bin/bash
cd ~/nous-chinese-analysis && python3 daily_service.py --run
EOF
chmod +x ~/.hermes/scripts/chinese-community-daily.sh

# 2. Create the cron job (runs daily at 9am Shanghai = 1am UTC)
hermes cron create "0 1 * * *" \
    --name "chinese-community-daily" \
    --script "chinese-community-daily.sh" \
    --no-agent \
    --workdir ~/nous-chinese-analysis

# 3. Verify
hermes cron list
python3 daily_service.py --status
```

The `--no-agent` flag means the script runs directly — no LLM session is spawned. The script's stdout is delivered to the `local` delivery target.

### Prerequisites

Before `--run` works against the live Feishu group, lark-cli must be re-authenticated:

```bash
/tmp/larkwrap.sh auth login --scope "im:message:readonly im:chat:read im:chat.members:read contact:user.base:readonly"
```

Test with: `/tmp/larkwrap.sh auth status`
