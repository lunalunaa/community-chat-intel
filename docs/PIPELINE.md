# Stream Processing Pipeline — Architecture & Operations

> **Scope:** Documents the multi-stream analysis pipeline (`semantic_retrieval`, `fact_extraction`, `deterministic_analytics`, synthesis, post-analysis) architecture. Concrete numbers below are from **one real example run** (a ~28K-message, 20-day chat export) — kept as illustrative figures. Swap in your own dataset's numbers when you run against a different community.  
> **Related:** The core keyword-analysis pipeline (`analyze.py`, `topics.py`, `crosstabs.py`) is documented in [README.md](../README.md). The research methodology rationale is in [plan.md](../plan.md).

---

## Architecture Overview

The pipeline runs four parallel **streams** against the same raw chat export, then synthesizes their outputs into a final report. Each stream answers different research questions using different technical approaches — no single method is reliable enough on its own.

```
                        chat_export.jsonl (raw NDJSON)
                                  │
                                  │  Stream A (topics.py)
                                  │  ─────────────────────────
                                  │  LLM topic classification
                                  │  12 categories × all msgs
                                  │  → topics.json
                                  │
                    ┌─────────────┼─────────────┐
                    │             │             │
              Stream B       Stream C       Stream D
              ─────────     ─────────     ─────────
              Semantic     Structured    Deterministic
              retrieval    fact extract  Python analytics
              BGE-M3 +     LLM per       (no LLM, free)
              FAISS +      ~100-msg      Pareto, stickiness,
              LLM          chunks        gateway mix, reply
              N queries    structured    graph, temporal
                    │             │             │
                    └─────────────┼─────────────┘
                                  │
                    narrative_synthesis.py
                    (single large-context LLM call)
                                  │
                        findings_final_v2.md
                                  │
                    build_final_report.py
                    (LLM-generated recommendations)
                                  │
                          final_report.md
                                  │
                    post_analysis.py
                    (brand audit, cost drill, CSVs)
```

**Why four streams?** Each compensates for another's blind spots:

| Stream | Strength | Blind spot |
|--------|----------|------------|
| **A** (topics) | Broad coverage; every message tagged | Coarse; can't catch sub-thread nuance |
| **B** (semantic) | Finds things keyword search misses; narrative synthesis | Query-dependent; misses if you don't ask |
| **C** (facts) | Structured, queryable, counts everything | LLM hallucination risk; schema bleed |
| **D** (deterministic) | Ground truth; no hallucination risk | Surface-level; can't interpret sentiment |

## Prerequisites

All streams assume:

1. **Raw chat export** as NDJSON at a known path (one JSON object per line). Produced by your platform's export tool (DiscordChatExporter, Telegram export, Lark CLI, etc.).
2. **An LLM CLI configured** with a working provider for LLM streams (A, B, C, synthesis). The scripts shell out to `hermes chat -q` by default — no direct API keys needed in this codebase. Swap the `cmd` list in each script if you use a different CLI.
3. **Python 3.10+** with `sentence-transformers`, `faiss-cpu`, `numpy` for Stream B embeddings.
4. **Salt file** for user-ID hashing (auto-generated by `parallax-analyze` on first run; default `./user_hash_salt.key`, override via `SALT_FILE`).

## Stream B — Semantic Retrieval + Narrative Synthesis

**File:** `semantic_retrieval.py`

### What it does

1. Embeds all non-system, non-trivial messages with `BAAI/bge-m3` (multilingual, supports 100+ languages)
2. Builds a FAISS `IndexFlatIP` (inner product = cosine similarity on normalized vectors)
3. Runs structured retrieval queries (from `QUERIES_BY_LANGUAGE`, selected via `TARGET_LANGUAGE`), each returning the top 30 most semantically-similar messages
4. For each query, feeds those 30 messages to an LLM to produce a ~150-200 word narrative synthesis

### Inputs

| Input | Source |
|-------|--------|
| `CHAT_JSONL` | Raw chat export (NDJSON) |
| Salt key | `./user_hash_salt.key` (or wherever `SALT_FILE` points) |

### Outputs

| File | Description |
|------|-------------|
| `stream_b/embeddings.npy` | Float32 embedding matrix |
| `stream_b/ids.json` | Message-ID ordering for the embedding array |
| `stream_b/retrieval_results.json` | Top 30 hits per query with scores |
| `stream_b/synthesized_findings.json` | Per-query narrative analyses (structured JSON) |
| `stream_b/findings.md` | Human-readable markdown version |

### Query sets

Queries are organized by `QUERIES_BY_LANGUAGE` — the `zh` and `en` sets ship as examples. Each set covers the same research shapes:

| Category | Example queries |
|----------|---------|
| **Messaging-adapter demand** | Platform-specific adapter demand + actively-building signals |
| **Gateway / hosted vs direct API** | Hosted service usage, direct vendor API, OpenRouter, key-sharing |
| **Topic texture** | Install friction, brand confusion, pricing complaints, network friction, success stories, feature requests, core feature usage |
| **Competitor sentiment** | Comparison with competitor products, comparison with alternative tools |
| **Community hubs** | Mentions of external communities |

Add your own `QUERIES_BY_LANGUAGE` entry for other languages, or edit the existing sets to match your product's feature names and competitors.

### Performance

- Embedding ~28K messages: ~2-5 minutes on CPU (single-threaded)
- Synthesis calls: ~10-15 minutes sequential; each call is independent
- Embedding cache: if `embeddings.npy` + `ids.json` exist and match, skip re-embedding

### Gotchas

- **Env vars** — set `CHAT_JSONL` and `OUT_DIR` before running
- **Embedding cache** — delete `stream_b/embeddings.npy` to force re-embedding after changing the input
- **Model availability** — synthesis calls use the LLM configured via `LLM_PROVIDER` / `LLM_MODEL` env vars. If unavailable, edit the `cmd` list in the script
- **Message truncation** — only first 1,500 chars of each message are embedded (to stay within BGE-M3's context window)

## Stream C — Structured Fact Extraction

**File:** `fact_extraction.py`

### What it does

1. Splits all content messages into windows of ~100 consecutive messages (preserving conversational order)
2. Feeds each chunk to an LLM with a JSON schema requesting 10 categories of structured facts
3. Aggregates facts across all chunks
4. Uses tolerant JSON parsing (trailing-comma stripping, json5 fallback) on every chunk — no separate retry pass needed

### The 10 fact categories

| Category | Output field | Example facts |
|----------|-------------|---------------|
| Install problems | `install_problems` | "Docker can't mount volume", "pip install fails on Windows" |
| Provider usage | `provider_usage` | "using a direct vendor API, sentiment=positive" |
| Messaging intent | `messaging_intent` | "wants_adapter for a messaging platform" |
| Brand confusion | `brand_confusion` | "is this the official site?" |
| API key sharing | `api_key_sharing_evidence` | "offering invite code for key-sharing" |
| Pricing complaints | `pricing_complaints` | "burns more tokens than a competitor" |
| Feature requests | `feature_requests` | "want one-click integration with X" |
| Success stories | `success_stories` | "built a bot with skills + cron" |
| Network friction | `vpn_network_friction` | "can't access from my region" |
| Competitor mentions | `competitor_mentions` | "Competitor A has better integration" |

### Inputs

| Input | Source |
|-------|--------|
| `CHAT_JSONL` | Raw chat export |
| Salt key | `./user_hash_salt.key` (or wherever `SALT_FILE` points) |

### Outputs

| File | Description |
|------|-------------|
| `stream_c/chunks_cache.jsonl` | Per-chunk raw LLM output (resume-safe; append-only) |
| `stream_c/aggregated_facts.json` | All facts across 10 categories |
| `stream_c/summary.json` | Aggregate counts + breakdowns |
| `stream_c/extraction_errors.json` | Chunks that failed JSON parsing (for retry) |

### Execution flow

```
fact_extraction.py
  ├── Load messages, sort chronologically
  ├── Chunk into windows of 100
  ├── For each chunk (concurrency=4):
  │     ├── Check cache → skip if done
  │     ├── Call LLM with schema prompt
  │     ├── Tolerant JSON parse (strict → strip commas → json5 → last-object)
  │     ├── Write to cache (append, flush)
  │     └── Log progress every 5 chunks
  ├── Aggregate all facts
  └── Write aggregated_facts.json + summary.json + extraction_errors.json

If extraction_errors.json is non-empty, those chunks failed even tolerant
parsing. Inspect the `_raw` field in the cache for manual review.
```

### Performance

- ~244 chunks × ~45s each (at concurrency=4): ~45 minutes total
- Cache is append-only and survives Ctrl-C — rerun picks up where it left off

### Gotchas

- **Trailing commas** — LLMs frequently add trailing commas in JSON arrays. The retry script handles this; the main script does not (failures go to `extraction_errors.json`)
- **Schema bleed** — facts occasionally land in the wrong category. Aggregates are trustworthy; per-record field-type purity is not
- **Env vars** — set `CHAT_JSONL` and `OUT_DIR` before running
- **Salt file** — independently reads `SALT_FILE` (doesn't import `analyze.py`)

## Stream D — Deterministic Python Analytics

**File:** `deterministic_analytics.py`

### What it does

Runs 6 pure-Python analyses with no LLM calls — instant, free, and hallucination-proof:

| Analysis | Output | What it measures |
|----------|--------|-----------------|
| **User Pareto** | `user_pareto_v4.json` | Message concentration: top-N posters' share of total messages, anchored to ground-truth membership |
| **User stickiness** | `user_stickiness_v4.json` | Retention breakdown: one-time posters vs single-day vs few-day vs multi-week (7+ days) active, as % of both posters and total human membership |
| **Temporal growth** | `temporal_growth_v4.json` | Day-by-day: new members joined, first-time posters, messages, active users, cumulative curves |
| **Gateway mix** | `gateway_mix.json` | URL-based classification of which API gateways/providers users reference |
| **Reply graph** | `reply_graph.json` | Thread depth stats: total replies, distinct threads, threads with 5+/10+ replies, longest threads |
| **Top users HTML** | `top_20_users.html` | Readable full-message timelines of the 20 most active posters (for qualitative review) |

### Why ground-truth anchoring

Earlier versions derived membership from system-message join events, but many platforms don't log leave/removal events. This caused membership overcounts. The current version fixes this by pinning to **live membership verified via the platform's admin UI**:

- Set `GROUND_TRUTH_HUMANS` and `GROUND_TRUTH_BOTS` env vars from the platform's member list
- This becomes the authoritative denominator for all retention and engagement rates

### Gateway URL classifier

URLs in message content are matched against domain patterns to classify gateway choice. The patterns are configurable — the shipped set covers common hosted services, direct vendor APIs, self-hosted setups, and proxy resellers. Edit the `GATEWAY_PATTERNS` dict in the script to match your ecosystem.

### Performance

- All 6 analyses combined: <5 seconds on a ~28K-message corpus
- No API calls, no cost, no rate limits

### Gotchas

- **Env vars** — set `CHAT_JSONL`, `OUT_DIR`, `GROUND_TRUTH_HUMANS`, `GROUND_TRUTH_BOTS` before running
- **Gateway classifier** — matches URLs in message text only. Users who talk about providers without pasting URLs won't show up here (Stream C covers those)
- **Reply graph** — only tracks explicit `reply_to` threading. Inline `@`-mentions and adjacent-message answers are invisible

## Cross-Stream Synthesis

**File:** `narrative_synthesis.py`

### What it does

Gathers outputs from **all four streams** + ground-truth anchors, builds a single large-context prompt, and feeds it to a stronger LLM model to produce `findings_final_v2.md` — a 14-section narrative survey.

### Sections produced

| # | Section | Primary source |
|---|---------|---------------|
| 1 | Executive Summary | All streams |
| 2 | Who Are They? | Stream D (membership ground truth) |
| 3 | What Do They Talk About? | Stream A (topic distribution) |
| 4 | Provider / Gateway Landscape | Stream C (provider_usage facts) |
| 5 | Messaging Platform Usage | Stream B + Stream C |
| 6 | Install & Setup Reality | Stream C (install_problems) |
| 7 | Competitor Landscape | Stream C (competitor_mentions) + brand audit |
| 8 | Brand Identity & Confusion | Stream C (brand_confusion) |
| 9 | API Key Sharing / Gray Market | Stream C (api_key_sharing_evidence) |
| 10 | Pricing / Token Consumption | Stream C (pricing_complaints) |
| 11 | Network / Geographic Friction | Stream C (vpn_network_friction) |
| 12 | Feature Requests & Success Stories | Stream C |
| 13 | Help Ecosystem Health | Stream D (reply graph) |
| 14 | Notable Quotes / Texture | Sampled exemplars from Stream C |

### Model selection

The synthesis and recommendations use the model configured via `LLM_MODEL_PRO` (a stronger variant). The base `LLM_MODEL` is used for per-message/chunk classification (Streams A, B, C) where throughput matters more than nuance.

### Gotchas

- **Context length** — the prompt is large (~36K characters in the example run). If adding more data, watch for token limits
- **Model availability** — the Pro variant may not always be available. Falls back to base model if needed (edit the `--model` flag)
- **Env vars** — `OUT_DIR` is used to locate stream outputs

## Build Final Report

**File:** `build_final_report.py`

### What it does

Assembles `final_report.md` from multiple components:

1. **Front matter** — header with metadata
2. **Executive Summary** — extracted from `findings_final_v2.md` §1
3. **Methodology** — written directly in the script (prose describing data collection, ground-truth anchoring, analysis streams, tools, privacy)
4. **Findings** — body of `findings_final_v2.md` (§§2-14, renumbered)
5. **Recommendations** — generated by a separate LLM call with explicit P0/P1/P2/Watch tiering instructions and structural context
6. **Limitations & Caveats** — written directly in the script
7. **Appendices** — file inventory with sizes, reproduction instructions

### The recommendations generator

The recommendations prompt is architecturally significant — it explicitly tells the model:
- The structural context relevant to your product (via `STRUCTURAL_CONTEXT` and `RECOMMENDATION_FOCUS_AREAS` env vars)
- To produce P0/P1/P2/Watch tiers with cost/effort/impact estimates
- To include a section on "What Is Explicitly NOT Recommended"
- To include open questions for leadership

This ensures the recommendations are grounded in the data rather than generic best-practices.

## Post-Analysis

**File:** `post_analysis.py`

### What it does

Three standalone drill-downs that run after the main synthesis:

#### (a) Brand Audit
- Regex-scans all messages against configurable brand/variant patterns (`BRAND_PATTERNS` in the script)
- Discovers patterns not already known (e.g., unofficial shadow-docs clusters, newly-surfaced competitor/clone names)
- Merges with Stream C `brand_confusion` incidents for resolution breakdown
- Output: `post/brand_audit.md` with counts, observations, and sampled messages

#### (b) Cost Complaint Drill
- Keyword-filters for cost/consumption complaints specifically about your product
- Pulls Stream C `pricing_complaints` incidents
- Output: `post/token_consumption.md`

#### (c) CSV Exports
- Exports Stream C structured facts as UTF-8 BOM CSVs (Excel-compatible)
- One CSV per fact category
- Output: `post/*.csv`

## Full Execution Order

When reproducing against a new dataset, run in this order:

```bash
# 0. Prerequisites — get the raw export
#    (Discord: DiscordChatExporter; Telegram: desktop export; Lark: @larksuite/cli)
#    Result: chat_export.jsonl (NDJSON, one JSON object per line)

# 1. Core pipeline (optional — establishes keyword baselines)
parallax-analyze --input chat_export.jsonl --platform lark --out ./out -v

# 2. Stream A — topic classification
parallax-topics \
    --input-chat chat_export.jsonl \
    --platform lark \
    --out ./out/topics.json

# 3. Stream D — deterministic analysis (run first; cheapest, catches data issues)
python -m parallax.streams.deterministic_analytics
#    ↑ set CHAT_JSONL, OUT_DIR, GROUND_TRUTH_HUMANS/BOTS env vars before running

# 4. Stream B — semantic retrieval + synthesis
python -m parallax.streams.semantic_retrieval
#    ↑ set CHAT_JSONL, OUT_DIR, TARGET_LANGUAGE env vars before running
#    Slowest step (embedding). Cache survives reruns.

# 5. Stream C — structured fact extraction
python -m parallax.streams.fact_extraction
#    ↑ set CHAT_JSONL, OUT_DIR, TARGET_LANGUAGE env vars before running
#    Tolerant JSON parsing is built in — no separate retry step.

# 6. Cross-stream synthesis
python -m parallax.streams.narrative_synthesis
#    → findings_final_v2.md

# 7. Post-analysis drills
python -m parallax.streams.post_analysis
#    → post/brand_audit.md, post/token_consumption.md, post/*.csv

# 8. Final report assembly
python -m parallax.streams.build_final_report
#    → final_report.md
```

### Incremental analysis

When you re-export the same chat periodically, use `--incremental` to only
process new messages and merge results into the existing `stats.json`:

```bash
# First run (full):
parallax-analyze --input export.json --platform discord --out ./out

# Later runs (incremental — only new messages are processed):
parallax-analyze --input export_updated.json --platform discord --out ./out --incremental -v
```

A SQLite state store (`parallax_state.db` in the output dir) tracks message
IDs. Counter-type stats are summed; user counts take the max (conservative).

### Comparing runs

```bash
parallax-diff --old ./run1/stats.json --new ./run2/stats.json
parallax-diff --old ./run1/stats.json --new ./run2/stats.json --json
```

Shows KPI deltas, per-counter changes (new/gone/changed), and percentage deltas.

## Adapting for a New Dataset

All stream scripts read configuration from environment variables. Before running:

1. **Set `CHAT_JSONL`** — path to the raw NDJSON export
2. **Set `OUT_DIR`** — output directory (creates `stream_b/`, `stream_c/`, `stream_d/`, `post/` subdirectories)
3. **Set `TARGET_LANGUAGE`** — which `QUERIES_BY_LANGUAGE` set to run, and how LLM prompts describe the corpus's language
4. **Set `TS_UTC_OFFSET_HOURS`** — UTC offset for display-format timestamps lacking a timezone marker (default `8`; set `0` for UTC, `9` for Japan/Korea, etc.)
5. **Set ground truth** (Stream D only) — `GROUND_TRUTH_HUMANS` and `GROUND_TRUTH_BOTS` from platform UI
6. **Verify salt file** — all scripts read `SALT_FILE` (default `./user_hash_salt.key` in the current working directory)
7. **Check model availability** — scripts shell out to `hermes chat -q` by default (see `LLM_PROVIDER`/`LLM_MODEL`/`LLM_MODEL_PRO` env vars); if your provider/model is down, override those env vars or edit the `cmd` list in the relevant script

### Quick-start env vars

```bash
export CHAT_JSONL=./data/chat_export.jsonl
export OUT_DIR=./out
export TARGET_LANGUAGE=zh        # or en, ja, ko, etc.
export TS_UTC_OFFSET_HOURS=8     # your export's local timezone offset
export GROUND_TRUTH_HUMANS=3119  # from platform admin UI
export GROUND_TRUTH_BOTS=5
```

## Key Design Decisions

### Why shell out to an LLM CLI instead of using APIs directly?

- **Zero auth surface:** No API keys in the scripts — all provider auth lives in your LLM CLI's own config
- **Provider portability:** Switch models by changing one env var
- **Rate limiting, retries, streaming:** Handled by the CLI, not reimplemented per-script

### Why Stream D anchors to live UI membership

Earlier versions derived membership from system-message join events. But many platforms don't emit system messages when users leave or are removed from a group. Churn is then visible only as a residual between join events and live membership; individual-user churn patterns are not recoverable without periodic snapshots. The live UI figure is the only reliable denominator for retention rates.

### Why Stream C uses per-chunk caching (append-only JSONL)

The chunk extraction takes ~45 minutes. If interrupted (Ctrl-C, network blip, model outage), the cache ensures only unprocessed chunks are rerun. The append-only design means even partial progress is never lost.
